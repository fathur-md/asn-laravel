## 2. What This Project Teaches About Leadership

### Ferdi's Role

Ferdi proposed the project idea, designed the features, and created the initial implementation. This is valuable — a vision is important. But leadership in a student project is not just about starting; it is about **sustaining participation across the entire team**.

**The hard lessons for a team lead:**

- **Setting expectations early.** Did each member explicitly agree to their module before coding began? Without a verbal or written contract, members can drift.
- **Creating accountability.** Did the lead follow up on progress daily? Did members report blockers? In a 7-day sprint, two days of silence is a crisis.
- **Enforcing Git discipline.** Allowing members to commit under another person's name, or to work on a shared laptop without separating contributions, destroys traceability. This is not a technical problem — it is a management problem.
- **Recognizing non-participation early.** By day 3, it was likely clear that some members were not contributing. Did the lead escalate? Did the team address it? Or was it easier to just absorb the work and hope the problem solved itself?

### What Good Leadership Looks Like in a Student Project

- **Daily standups** — even via WhatsApp or Discord, 5 minutes each: "What did you do? What will you do? Any blockers?"
- **Feature branches** — each member works on their own branch. If a branch has no commits after 48 hours, the lead intervenes.
- **Pairing, not absorbing** — when a member is stuck, pair with them rather than doing their work for them.
- **Honest reporting** — if the contribution report submitted to the lecturer has to rely on "shared laptop" explanations, the leadership process failed.

## 3. Dealing with Inactive or Low-Skill Members

This is the most realistic and painful part of any student group project. Here are the patterns observed in this project and how they should have been handled:

### The Problem

- **Member with no skill:** Does not know Laravel, Git, or basic PHP. They freeze, contribute nothing, and hope the group carries them.
- **Member who is inactive:** Simply does not respond, does not show up, does not commit. No explanation given.
- **Member who is lazy:** Has the skill but lacks the motivation. They produce late, low-quality work that the group has to rewrite.

### How the Team Handled It (Based on Evidence)

- Julius has 1 commit. Enrico has 0. Both were explained as "shared laptop with Ferdi."
- The actual productive work was redistributed to Fathur and Maria.
- The project got finished, but at the cost of burnout for the active members and zero learning for the inactive ones.

### How It Should Be Handled

| Strategy          | Action                                                                                                                                                                                                                                                       |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Skill gap**     | Assign non-coding tasks: documentation, testing, deployment research, screenshot creation, README writing, slide preparation. Everyone can contribute something.                                                                                             |
| **Inactivity**    | Set a hard deadline for first commit (e.g., within 24 hours of repo creation). If no response in 48 hours, escalate to the lecturer. Do not carry dead weight silently.                                                                                      |
| **Laziness**      | Assign the smallest possible concrete task with a 12-hour deadline. If they deliver, build from there. If not, the problem is now documented and can be escalated.                                                                                           |
| **Shared laptop** | Never allow this. Each member must commit from their own GitHub account, on their own machine, or from GitHub Codespaces. If they share a laptop, at minimum they must use separate Git user configurations or separate browser sessions for GitHub commits. |

### The Hard Truth

In the real world, a team member who contributes nothing gets fired. In a student project, you cannot fire anyone. But you **can document everything** — who did what, when, and the evidence. A lecturer cannot fairly grade a member who has 0 commits unless the team never reported the issue.

## 7. Final Reflection

This project produced a functional, well-crafted application. The code is clean, the architecture follows Laravel conventions, and the documentation is thorough. **On a technical level, the team succeeded.**

But a student project is not just a code deliverable. It is a **learning exercise in collaboration**. And on that dimension, there are clear gaps.

The best teams are not the ones with the most talented individuals. The best teams are the ones where:

- Every member can explain every part of the project
- Git history shows balanced contributions from all members
- The leader creates space for participation, not just a finished product
- Problems are addressed openly, not absorbed silently

**To the lecturer:** The requirements in TUGAS.md were clear. The requirement for every member to commit was not met. Whether the shared-laptop explanation is accepted is a judgment call, but it should not be treated as equivalent to actual individual commits from each member's own account.

**To the team:** Take this experience forward. The skill of building a Laravel CRUD is useful. But the skill of building a **functional, fair, and psychologically safe team** is far more valuable — and it transfers to every project you will ever work on.
